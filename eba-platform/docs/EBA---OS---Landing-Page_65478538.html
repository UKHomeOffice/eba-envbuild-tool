Introduction

The Environment Build Automation (EBA) tooling has been developed to enable an environment to be designed using Visio and to be automatically provisioned in the cloud. Two components of the tooling, EBA Visio Physical Design and EBA ACloud, have been open sourced and together enable environments designed in Visio to be provisioned in the cloud. The virtual machines in that network will have an operating system image but no additional middleware or user applications loaded and configured. The programme that gave rise to these tools has a more comprehensive toolset that does have capabilities to do orchestrated middleware and application deployment but is not open sourced at this time. By way of example of how these tools can form part of a more comprehensive environment build toolset the environment build tooling design that these tools are being used in is outlined here.

The basic premise is that a physical environment design is composed in Visio using the tools provided in the EBA Visio Physical Design project. This provides a custom Visio stencil containing the custom shapes and associated shape data, a configuration file and a programme to enable the design to be exported to an XML file conforming to an EBA provider agnostic schema. The environment definition XML, along with a Build Plan XML is fed into the EBA ACloud Client tool which then provisions your environment in your cloud provider (AWS or a VMware based cloud provider that uses the VMware vCloud Director (v5.1 up to v5.6) API's, e.g Skyscape).

EBSA Environment Build Tooling

EBSA has developed a set of capabilities to perform Environment Build and Management, Environment Access Management and Continuous Integration. Some of the assets developed for that programme are being open sourced (the ACloud parser and Visio tool above) and others (the capacities below) have been packaged for restricted distribution.

 

The environment build and management tooling includes the following capabilities:

  • Automated build and modification of Skyscape or AWS hosted resources (virtual applications, virtual machines and networks) using the vCloud Director or AWS E2C API based upon configuration managed meta data (XML definition files) and operating system templates
  • Automated layering and modification of the following configuration managed meta data and deployment artifacts onto the virtual machines:
    • Operating system configuration
    • Operating system patching
    • Middleware installation
    • Middleware configuration
    • Middleware patching
    • Application code installation (onto Middleware)
    • Application code configuration

  • A Self Service capability for the deployment and configuration of components, applications or full releases:

    • Standard UI for deployers: Ability for non-specialist, authorised individuals to easily and frequently deploy packaged applications into their own environments, including production environments.
    • Standard deployment interface for developers
    • Deployments are remote: Console access to target servers is not available to deployers at deployment time (ssh sudo access is uncommon)
    • Deployments are
      • robust
      • repeatable
      • fast
      • easy
