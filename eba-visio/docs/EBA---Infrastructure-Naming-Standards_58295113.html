EBA - Infrastructure Naming Standards

Overview

The following page outlines the infrastructure naming standards suggested to be used by users of the EBA tools.  The following infrastructure resources have standards which govern their names:

Standards

1 - VPC (Virtual Private Cloud) name

The VPC is an AWS specific concept.

Format

<Account shortname>_<Programme_shortname>_<Environment container shortname>_<Project name>_<Environment shortname + number>

Usage

Attribute
Necessity 
Description
Account shortnamemandatory

An abbreviation of the name of the account that the virtual application belongs to - for example “HO” (abbreviated from Home Office).

Programme shortnameoptionalAn abbreviation of the programme of work within the account that the virtual application belongs to.  Where the virtual application is used across multiple programmes, then this abbreviation will be omitted - this will be set to "IPT" for virtual applications used exclusively by the IPT programme.
Environment container shortnamemandatory

An abbreviation of the environment container type that the virtual application belongs to. The valid set of environment container shortnames are shown within the shortname values table below.

Project nameoptionalAn abbreviation of the project that this environment belongs to. The valid set of project shortnames are shown within the shortname values table below.
Environment shortname + numbermandatory

An abbreviation of the name of the development or test environment that this virtual application belongs to. For tooling environments (including their boundary controls), a specific tooling shortname will be used as they span all development and test environments within the environment container. The valid set of environment shortnames are shown within the shortname values table below.

N.B. Each environment shortcode is post-fixed with a number (beginning at 1) to enable multiple instances of the same environment to be built and hosted alongside each other e.g. PRP1 and PRP2 are different instances of the same environment used for different test activities. All tooling environments (including their boundary controls) will use the same number, unless second instances of those environments are created.

Examples

The following examples demonstrate the use of this standard:

  • HO_IPT_NPA_SIT1 -  a VPC used by the System Integration Test 1 (SIT1) environment, within the Non-Production AWS (NPA) environment container of the Immigration Platform Technologies (IPT) project for the Home Office (HO) account
  • HO_IPT_NPA_TLG1 - a VPC used by the Tooling (TLG) environment, within the Non-Production AWS (NPA) environment container of the Immigration Platform Technologies (IPT) project for the Home Office (HO) account
  • HO_IPT_NPA_II_DAT1 - a VPC used by Immigration Identity Assurance System, Development and Test 1 (DAT1) environment, within the Non-Production AWS (NPA) environment container of the Immigration Platform Technologies (IPT) project for the Home Office (HO) account.

6 - Virtual application template name

 Note that this naming standard only applies to your company/environment containers created vApp templates.  If a template is supplied from a 3rd party (e.g. Brocade, Fortigate) it should be uploaded with the same name as the upstream provider uses.

Format

<Your company/environment container>-TEMPLATES-<Operating System>-<Operating System Version>-<Operating System Architecture>

Usage

N.B. No full stops should be used in versions as this prevents the templates being uploaded or downloaded. They are replaced by underscores.

AttributeNecessity Description
Operating Systemmandatory

The name of the operating system.

N.B. This is all in uppercase

Operating System Versionmandatory

The version of the operating system e.g. 5_5

Operating System ArchitecturemandatoryThe architecture of the operating system e.g. 64

Examples

 The following examples demonstrate the use of this standard for the ACME company/environment container:

  • ACME-TEMPLATES-CENTOS-6.5-64 - a CentOS 6.5, 64 bit operating system template

 

Note: in rare instances (usually development/test only e.g. Security PoC) virtual appliances packaged as complete images including the operating system will be supplied for installation as .ovf files, so these will not have specific OS details.

9 - Short host name

When creating a virtual machine, the 'Virtual Machine name’ and the ‘Computer name’ properties will be populated with a fully qualified domain name, comprising of a short host name and a domain suffix.  The fully qualified domain name and short host names are also registered within environment container and virtual application DNS respectively to enable each interface from a virtual machine host to be referenced by name.

Two types of DNS entries are registered:

  1. Environment container DNS - the fully qualified domain names of each network interface from a virtual machine are registered to an DNS server (within the management tooling zone) within an environment container to enable virtual machines across different virtual applications to talk to each other by fully qualified domain name using explicit network interfaces (this enables a fully qualified domain name to be resolved by the environment container DNS to a specific network card interface on a virtual machine within a different virtual application to the calling virtual machine).
  2. Virtual Application DNS - the domain shortnames of each network interface from a virtual machine are registered to a DNS server (on the management router of the virtual application) within the virtual application to enable virtual machines within the virtual application to talk to each other by short host name using explicit network interfaces (this enables a short host name to be resolved by the virtual application DNS to a specific network card interface on a virtual machine within the same virtual application as the calling virtual machine).

N.B. The short host name used as part of the fully qualified domain name within the virtual machine name / computer name will always be the same short host name used on that machines management network interface even if the virtual machine does not have a management network card.

N.B. Host names must be unique within a domain (virtual application) as they are used within a virtual application to reference other hosts within that virtual application.

The role of a host enables similar types of virtual machine to be categorised together to enable common configuration to occur. The valid set of virtual machine roles are documented within the VM Role Identifiers page.

Security compartment

4

A single lowercase character starting at 'a' and then allocated in sequence through the alphabet to each security compartment within a zone or boundary control.

N.B. Routers are classified within their own security compartment in this context:

  • Within a boundary control, the north router / firewall is assigned 'a', the south router is assigned 'b' (if present), the management router is assigned 'c' (if present), with subsequent letters being allocated to each security compartment within that boundary control, starting from 'd'.
  • Within a zone, the shared north / south router is assigned 'a', the management router is assigned 'c' (if present), with subsequent letters being allocated to each security compartment within that zone, starting from 'd'.
  • There have been events where within a zone, there are no security compartments; all virtual machines are therefore assigned 'a'.  However, to remove ambiguity, all zones will now contain security compartments starting at 'e', unless there is a management proxy where the security compartments will start at 'd'.
Network traffic type5

A single lowercase character abbreviation representing the type of network traffic that the interface with this host name is used for. The valid abbreviations are:

  • a - application common interface
  • n - application north interface
  • s - application south interface
  • m - management interface

N.B. Within a zone a common router is used, in this instance a virtual machine would have an application common interface (used for north and south), and a management interface. Whereas, the common router within the zone, would have an application north interface (to the less trusted zone) and (if applicable) an application south interface (to the more trusted zone).

Instance id

6 to 7

(2 chars)

This field has been overloaded for two purposes:

  1. To represent a virtual IP address (VIP), used to load balance across a number of network interface card IP addresses.
  2. To represent a numbered instance of a type of host.

The use of this field for each purpose is described below:

  1. A single lowercase character 'v' preceded by an alphanumeric from 1 to 9, a-z e.g. 1v  or  av
  2. An alphanumeric character followed by a single digit number starting at 01, N.B. the need for the preceding '0' a cluster of two SOA Suite hosts would be numbered 01 and 02, a secondary cluster would be numbered 11 and 12 even if all numbers 01-09 have not been used. e.g. first cluster has two VMs 01 and 02, second cluster has three VMs 11, 12, and 13. After the 9x range is used then switch to alphabetics a1, b1 etc.

Examples

The following examples demonstrate the use of this standard:

  • wpxdm01 - the short host name of the first instance (01) of a web proxy host (wpx) within security compartment d (within a security zone represented by the virtual application), which is used within DNS to identify the management network interface (m) of the host.
  • soaea02 - the short host name of the second instance (02) of a SOA suite host (soa) within security compartment e (within a security zone represented by the virtual application), which is used within DNS to identify the application (common) interface (a) of the host.
  • roubs01 - the short host name of the first instance (01) of a router host (wpx) within security compartment b (within a boundary control represented by the virtual application), which is used within DNS to identify the application south network interface (s) of the host.

  • HO_IPT_NPA_INT1 - a Edge Gateway router, connected externally to the Internet, within an environment container used for environments with non-live data, by the IPT programme within the Home Office account

 


Common Data Values

The following table outlines the valid set of values for each shortname used within the infrastructure hosting standards:

ShortnameValid Values

Environment

A sample set of environment shortname values are listed below :

  • DEV - Development
  • CIT - Component Integration Test
  • SIT - System Integration Test
  • TLG - Tooling
Network connection type

A sample set of network connection type shortname values are listed below:

  • I - An environment container isolated network that is not connected to the Edge Gateway router
  • R - An environment container router network that is connected to the Edge Gateway router
Network external connectivity

A sample set of external network connectivity shortname values are listed below:

  • INT - Internet
  • PSN - Public Services Network
Network traffic type

A sample set of network traffic type shortname values are listed below:

  • A - Application traffic between boundary controls / security zones (only used with isolated and virtual application networks)
  • M - Management traffic (only used with isolated and virtual application networks)
Environment container

A sample set of environment container shortname values are listed below:

  • NP - Non-Production environments
  • NPA - Non-Production environments hosted in AWS
  • PR - Production environments
 Programme shortname

 A sample set of programme shortname values are listed below:

  • IPT - IPT Programme
  • ABC - ABC Programme
Project name

A sample set of project letter identifiers are listed below:

  • II - II Project
  • XY - XY Project
Security compartment

A sample set of security compartment shortname values are listed below:

  • A - Security compartment A
  • B - Security compartment B
  • C - Security compartment C
  • n - Security compartment n (i.e. continuation of using alpha characters for subsequent compartments)
Virtual application

A sample set of virtual application shortname values are listed below:

  • CIZ<nr> - Continual Integration Zone
  • COBC - Core Boundary Control
  • CTZO - Collaborative Tooling Zone
  • DABC - Data Boundary Control
  • DAZO - Data Zone
  • DEZ<nr> - Development Zone
  • DTZO - Development Tooling Zone
  • EMZO - Environment Management Tooling Zone
  • ESZO - External Simulator Zone
  • INBC - Internet Boundary Control
  • PRZO - Private Application Zone
  • MABC - Management Boundary Control
  • MEBC - Management Edge Boundary Control
  • MTZO - Management Tooling Zone
  • SSZ<nr> - System Test Zone
  • TTZO - Test Tooling Zone
