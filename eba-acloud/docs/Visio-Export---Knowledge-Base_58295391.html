Visio Export - Knowledge Base

This page captures the complete set of validation errors that are produced from the EBSA Physical Design export process from Visio, and documents a recommendation on how to fix them.  It also captures any known issues that can occur, to enable them to be avoided.

Validation Errors

Error CodeError MessageSolution
001<Page> has no Environment Container.Each design must contain an Environment Container which indicates what kind of environment this design reflects. The shape needs to be positioned at the top of the design, above the Virtual Machine Container.
002<File>  has no pages with names containing '<Letters>'.Each design should consist of at least 2 pages. A version page which contains a log of changes made to the design and an actual design page with a title containing letters "<Letters>" in. E.g. HO_IPT_NPA_PRP3_DAZO
003<Page> does not state the environments provider.As well as having the Environment Container in the Design, it's also important to ensure that all of the shape data is filled out correctly. You can specify the environment provider by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate provider from the drop down list.
004<Page> does not state the environments name.As well as having the Environment Container in the Design, it's also important to ensure that all of the shape data is filled out correctly. You can specify the environment name by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate name from the drop down list.
005<Page> is missing the 'dataCentreName' property from the 'Environment Container'This is a AWS specific error indicating that the dataCentreName must be provided in order for the environment container to be processed correctly. You can specify the dataCentreName by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate data centre name from the drop down list.
006<Page> is missing the 'cidr' property from the 'Environment Container'This is a AWS specific error indicating that the cidr must be provided in order for the environment container to be processed correctly. You can specify the cidr by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window specifying appropriate cidr in the text field.
007<Page> has an incorrect environment name for AWS This is a AWS specific error indicating that the environment name you're trying to use is not applicable. At the moment only NPA can be exported to AWS. You can specify the environment name by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate name from the drop down list.
008<Page> has an incorrect environment name for SkyscapeThis is a Skyscape specific error indicating that the environment name you're trying to use is not applicable. At the moment only NP,ST and PR can be exported to Skyscape. You can specify the environment name by right clicking on the Environment Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate name from the drop down list.
009More than one environment container exists - <Name>You may only have a single environment container per design. Ensure that there's only a single environment container shape per design.
010<Page> has no Virtual Machine Container serviceLevel.This is a Skyscape specific error indicating that the serviceLevel must be provided in order for the virtual machine container to be processed correctly. You can specify the dataCentreName by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate service level from the drop down list.
011AWS does not require a Virtual Machine Container serviceLevel.Contrary to the error above AWS does not require serviceLevel to be provided. You can clear the serviceLevel by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window selecting empty field from the drop down list.
012<Page> is missing the 'domain' property from the 'Virtual Machine Container'As well as having the Environment Container in the Design, it's also important to ensure that all of the shape data is filled out correctly. You can specify the domain by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window specifying appropriate domain in the text field.
013<Page> is missing the 'name' property from 'Virtual Machine Container'As well as having the Environment Container in the Design, it's also important to ensure that all of the shape data is filled out correctly. You can specify the name by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window specifying appropriate name in the text field.
014<Page> is missing the 'nameDisplay' property from <Machine>. Remember to reflect the number of VMs.It's important that this text field is not blank and filled out correctly. Please also ensure that you follow the convention by specifying the number of VMs that are in this cluster. So for example if you have a pair of VMs, the name should be something like simdb01/02 (5 letters followed by numbers separated by slashes). You can specify the nameDisplay by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window specifying appropriate nameDisplay in the text field. Note that if the nameDisplay is blank, the error message will not tell you which virtual machine needs verifying therefore you'll have to search for it manually.
015<Page> is missing the 'description' property from <Machine>It's important that this text field is not blank and filled out correctly. You can specify the description by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window specifying appropriate description in the text field.
016<Page> is missing the 'templateName' property from <Machine>It's important that this text field is not blank and filled out correctly as this indicates which operating system image to use. You can specify the templateName by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window selecting appropriate template name from the drop down list.
017<Page> is missing the 'cpuCount' property from <Machine>It's important that this text field is not blank and filled out correctly as this indicates the amount of CPUs to use on this particular machine. You can specify the cpuCount by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window selecting appropriate cpu count from the drop down list.
018<Page> is missing the 'memory' property from <Machine>It's important that this text field is not blank and filled out correctly as this indicates the amount of RAM to use on this particular machine. You can specify the memory by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window selecting appropriate RAM amount from the drop down list.
019<NumberOfIPs> static IP addresses have been defined for <NetworkCard> network interface card but there are <NumberOfVms> VMs.Either the amount of static IP addresses specified or the name of the Virtual Machine is incorrect. Please also ensure that you follow the convention by specifying the number of VMs that are in this cluster. So for example if you have a pair of VMs, the name should be something like simdb01/02 (5 letters followed by numbers separated by slashes). And therefore in this case should only have 2 static IP addresses specified e.g. 10.0.0.1/2 (note the IPs are seperated by slashes similarly to the name, it's not a network mask/subnet range). You can verify these text fields by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window checking that both of the fields contain correctly formatted information.
020<NetworkCardName> is missing the 'staticIpAddress' propertyIt's important that this text field is not blank and filled out correctly as this indicates the IP Addresses to use for these virtual machines. You can specify the staticIpAddress by right clicking on the Network Card Interface, going to "Data" > "Shape Data..." and in the new window specifying appropriate IP Addresses in the text field.
021<NetworkCardName> is missing the 'ipAssignment' propertyThis is a Skyscape specific error indicating that the ipAssignment must be provided in order for the network card interface to be processed correctly. You can specify the ipAssignment by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window selecting appropriate ip assignment type from the drop down list.
022<NetworkCardName> is missing the 'VRRP' propertyThis is a Skyscape specific error indicating that the VRRP must be provided in order for the network card interface to be processed correctly. You can specify the VRRP by right clicking on the Virtual Machine Container, going to "Data" > "Shape Data..." and in the new window specifying appropriate VRRP in the text field.
023<NetworkCard> is not connected to a VM.This error indicates that this specific network card is not attached to any of the Virtual Machines. Please ensure that the line going into Virtual Machine is properly attached to the connection point (green square). Also it's worth noting that if the network card is being attached to the cluster/pair of VMs, it should only connect to the top layer within the group. If there are no connection points in the place where you would like to attach it to, you might need to create one by using the "Connection Point Tool" (Shift+Ctrl+1). Before creating a connection point please ensure that you select the virtual machine (not the whole group) you're trying to create a connection point on first. Otherwise the connection point will belong to the shape that you have currently selected and will not be processed correctly.
024<NetworkCard> is not connected to a network.This error indicates that this specific network card is not attached to any of the Networks. Please ensure that the line going into Virtual Machine is properly attached to the connection point (green square) on the Network that you're attaching it to. If there are no connection points in the place where you would like to attach it to, you might need to create one by using the "Connection Point Tool" (Shift+Ctrl+1). Before creating a connection point please ensure that you select the network you're trying to create a connection point on first. Otherwise the connection point will belong to the shape that you have currently selected and will not be processed correctly.
025<Network> is missing the 'name' propertyIt's important that this text field is not blank and filled out correctly. You can specify the name by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the name of the network in the text field.
026<Network> is missing the 'description' propertyIt's important that this text field is not blank and filled out correctly. You can specify the description by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the description of the network in the text field.
027<Network> is missing the 'primaryDNS' propertyIt's important that this text field is not blank and filled out correctly. You can specify the primaryDNS by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the primaryDNS in the text field.
028<Network> is missing the 'staticIpPool' propertyIt's important that this text field is not blank and filled out correctly. You can specify the staticIpPool by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the staticIpPool in the text field.
029<Network> is missing the 'networkMask' propertyThis is a Skyscape specific error indicating that the networkMask must be provided in order for the network to be processed correctly. You can specify the networkMask by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window selecting appropriate network mask from the drop down list.
030<Network> is missing the 'gatewayAddress' propertyThis is a Skyscape specific error indicating that the gatewayAddress must be provided in order for the network to be processed correctly. You can specify the gatewayAddress by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the gateway address in the text field.
031<Network> is missing the 'ipRangeStart' propertyThis is a Skyscape specific error indicating that the ipRangeStart must be provided in order for the network to be processed correctly. You can specify the ipRangeStart by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the ip range start in the text field.
032<Network> is missing the 'ipRangeEnd' propertyThis is a Skyscape specific error indicating that the ipRangeEnd must be provided in order for the network to be processed correctly. You can specify the ipRangeEnd by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window specifying the ip range end in the text field.
033Network type missing for network <NetworkName>It's important that this text field is not blank and filled out correctly. You can specify the network type by right clicking on the Network, going to "Data" > "Shape Data..." and in the new window selecting appropriate network type from the drop down list.
034The VM <VirtualMachine> was not found inside the VM container.This error indicates that the amount of VMs in the group/pair do not match the nameDisplay of the first VM. So for example if the name of the VM is dbsim01/02/03, there must be 3 VMs in the group for them to be processed correctly (reference to the error below for more detail).
035<VirtualMachine> name(s) format is invalid.This error indicates that the nameDisplay provided for the VM does not follow correct pattern. Please ensure that the name provided starts with 5 letters, followed by the numbers of each VM (depending on how many there are in the group). So for example the name: dbsim01/02 would indicate that the name of the boxes is dbsim and there are 2 of them in the group. Similarly dbsim01/02/03/04 would indicate that there are 4 boxes in the group. You can specify the nameDisplay by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window specifying the nameDisplay in the text field.
036<IpAddress> IP address(es) format is invalid.This error indicates that the staticIpAddress provided for the VM does not follow correct pattern. Please ensure that you provide appropriate number of IPs depending on the number of VMs that are in the same group. So for example if the name of the VM is dbsim01/02/03 this would indicate that there are 3 VMs in the group and therefore 3 IP address need to be provided e.g. 10.11.12.13/14/15 (seperated by slashes). You can specify the staticIpAddress by right clicking on the Virtual Machine, going to "Data" > "Shape Data..." and in the new window specifying the static ip address in the text field.
037No Connections to/from <Shape>.This error indicates that a network interface has been found, however it's not connected to any Networks or Virtual Machines. Please ensure that both ends of the connector are attached properly.

Known Issues

Known IssueSolution

There's a bug in Visio where sometimes certain shapes (VMs, NICs, Heartbeats) are not being exported properly. There's no known issue that's causing it therefore there is no permanent fix at the moment.

This is no longer an issue in the new version (re-write).

The workaround is to delete that specific shape that is not being exported into XML and to simply re-create it using the same data. It's also worth manually going through the XML file to make sure that there aren't any other similar shapes that are being ignored by the Visio.

In some cases the Network Interfaces in the XML are pointing to the wrong type of Network. Normally a VM would have 2 NICs. One of them would point to the Management network, the other to the Application Network. However there have been cases where both of the NICs would point to the Management network even though it appears that it's connected to the Application network.

This is no longer an issue in the new version (re-write).

This is because when the connection point (snap point) was created, the wrong network was selected and even though it appears that you're attaching the connection point to the right network, actually you are creating a connection point for the shape that you have currently selected. The solution is to simply delete the connection point (Shift+Ctrl+1), to select the correct network and to re-create the connection point. Do not forget to re-attach the NIC to the newly created connection point.

Some of the VMs in a cluster/pair would have different specification even though they're grouped together.

This is no longer an issue in the new version (re-write).

This is because the data has been explicitly specified on the other layers in the group. This would then require that all of the layers (virtual machines) in the group are updated with the same information. To avoid this issue, only fill out the top layer (first virtual machine) that you can clearly see in the group and leave the rest of them blank. The code should be able to retrieve all the information it needs from that first shape and apply it to the rest of them.
The changes made to the shape (Virtual Machine/NIC/Network) do not reflect in the XML.This could be because only the actual label (text on the shape) was modified and not the actual data sheet that contains all of the properties. When making changes make sure to modify the data sheet first and only change the label afterwards if necessary (in most cases the label will update itself when the data sheet is changed).
Version number of the XML is lower than the one currently in the repository.This could be because when the previous export was made, the person who carried it out did not commit their design .vsd file to the repository. Also it's worth mentioning that you should ensure that you sync all of the files before making any changes as Git is unable to merge .vsd changes (as it is not in plain text format), meaning that only one person should be modifying the design at a time.
